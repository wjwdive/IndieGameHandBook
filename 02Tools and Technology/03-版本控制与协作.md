# 版本控制与协作

## 概述

本章将介绍版本控制系统（Git）的基础知识、团队协作工具、分支策略和自动化流程，帮助你和团队高效管理游戏开发项目。版本控制是专业游戏开发的核心实践，它不仅是代码管理工具，更是团队协作、版本追踪和项目安全的基石。

**学习目标**：
- 理解版本控制在游戏开发中的重要性
- 掌握Git的基本操作和工作流
- 学习游戏开发中特有的版本控制挑战和解决方案
- 掌握团队协作工具和最佳实践
- 能够为项目建立高效的版本控制和协作流程

## 1、为什么版本控制对游戏开发至关重要

### 1.1 版本控制的基本价值

版本控制系统（Version Control System, VCS）是记录文件内容变化，以便将来查阅特定版本修订情况的系统。对于游戏开发来说，版本控制提供：

- **历史记录**：追踪谁在什么时候修改了什么
- **撤销能力**：安全地回退到之前的版本
- **分支管理**：并行开发不同功能而不互相干扰
- **团队协作**：多人同时工作在同一个项目上
- **备份保护**：代码和资产的安全存储

### 1.2 游戏开发的特殊挑战

游戏开发项目与一般软件项目在版本控制上有显著差异：

| 挑战 | 影响 | 解决方案 |
|------|------|---------|
| **二进制文件多** | 美术资源、音频文件、动画等 | Git LFS、Perforce等专门工具 |
| **文件体积大** | 纹理、模型、视频文件占用大量空间 | 大文件存储优化 |
| **协作复杂度高** | 美术、策划、程序需要不同工作流 | 精细的权限管理和工作流 |
| **版本管理复杂** | 需要管理游戏版本、资源版本、代码版本 | 语义化版本、标签系统 |

### 1.3 如果没有版本控制...

> **警告**：不使用版本控制进行游戏开发，就像在没有安全网的情况下走钢丝。

常见灾难场景：
- **"我的硬盘坏了，所有工作都丢了"**
- **"我不小心删除了重要文件，找不回来了"**
- **"我和队友同时修改了同一个文件，他的修改覆盖了我的"**
- **"我不知道这个bug是什么时候引入的"**
- **"我需要退回两周前的版本，但不知道哪些文件该恢复"**

## 2、Git基础与核心概念

### 2.1 Git与其他版本控制系统

| 系统 | 特点 | 适用场景 |
|------|------|---------|
| **Git** | 分布式，速度快，分支灵活 | 开源项目、小型团队、代码为主的项目 |
| **Subversion (SVN)** | 集中式，简单易用 | 传统企业、简单工作流 |
| **Perforce (Helix Core)** | 企业级，二进制文件优化 | 大型游戏工作室、AAA项目 |
| **Plastic SCM** | 专为游戏优化，可视化 | Unity项目、游戏开发团队 |

**独立开发者推荐**：Git + Git LFS（大文件存储）

### 2.2 Git核心概念理解

**工作区、暂存区、仓库**
- **工作区**：你电脑上看到的项目文件
- **暂存区**：准备提交的文件临时存放区
- **仓库**：存储项目历史记录的地方

**分支与合并**
- **分支**：从主线分离的独立开发线
- **合并**：将分支的修改整合回主线
- **冲突**：多人修改同一文件同一位置时发生

**远程仓库**
- **远程仓库**：托管在服务器上的项目副本
- **推送**：将本地提交上传到远程仓库
- **拉取**：从远程仓库下载最新修改

### 2.3 Git基本命令速查

```bash
# 初始化新仓库
git init

# 克隆现有仓库
git clone https://github.com/用户名/项目名.git

# 查看状态
git status

# 添加文件到暂存区
git add 文件名
git add .  # 添加所有修改

# 提交更改
git commit -m "提交信息描述"

# 查看提交历史
git log
git log --oneline --graph  # 简洁图形化显示

# 创建分支
git branch 分支名
git checkout -b 分支名  # 创建并切换到新分支

# 合并分支
git checkout main
git merge 分支名

# 与远程仓库交互
git pull  # 拉取远程更新
git push  # 推送本地提交
```

## 3、游戏开发中的Git实践

### 3.1 Git LFS：大文件处理

游戏开发中最大的挑战是处理大型二进制文件。Git LFS（Large File Storage）是Git的扩展，专门解决这个问题。

**安装和配置Git LFS**：
```bash
# 安装Git LFS
git lfs install

# 跟踪特定类型的文件
git lfs track "*.psd"
git lfs track "*.fbx"
git lfs track "*.wav"

# 查看跟踪规则
git lfs track

# 提交跟踪规则
git add .gitattributes
git commit -m "添加Git LFS跟踪规则"
```

**推荐的跟踪文件类型**：
```bash
# 图像文件
*.psd, *.tga, *.bmp, *.exr

# 3D模型
*.fbx, *.obj, *.blend

# 音频文件
*.wav, *.mp3, *.ogg

# 视频文件
*.mp4, *.mov, *.avi

# 其他游戏资产
*.unity, *.prefab, *.asset
```

### 3.2 .gitignore文件配置

为游戏项目创建合适的.gitignore文件，避免将临时文件、构建产物等提交到仓库。

**Unity项目.gitignore示例**：
```gitignore
# Unity临时文件
[Tt]emp/
[Oo]bj/
[Bb]uild/
[Bb]uilds/
[Ll]ogs/
[Uu]ser[Ss]ettings/

# Unity项目设置
*.csproj
*.sln
*.suo
*.tmp
*.user
*.userprefs

# 资产导入缓存
[Ll]ibrary/
*.meta

# 操作系统文件
.DS_Store
Thumbs.db

# 构建产物
*.apk
*.ipa
*.app
```

**Godot项目.gitignore示例**：
```gitignore
# Godot特定文件
.godot/
export_presets.cfg

# 导入资源
*.import

# 日志和缓存
logs/
tmp/

# 编辑器设置
editor_settings.tres
```

### 3.3 分支策略和工作流

**独立开发者简单工作流**：
```
main（主分支，稳定版本）
  ↑
feature/新功能（功能开发分支）
```

**小团队Git Flow变体**：
```
main（主分支，发布版本）
  ↑
develop（开发分支，集成测试）
  ↑
feature/功能1（功能A开发）
feature/功能2（功能B开发）
hotfix/紧急修复（线上bug修复）
```

**游戏开发推荐的提交信息格式**：
```
类型(范围): 简要描述

详细说明（如果需要）

- 变更点1
- 变更点2
- 变更点3

关联Issue：#123
```

**常用类型**：
- `feat:` 新功能
- `fix:` Bug修复
- `art:` 美术资源更新
- `sound:` 音频更新
- `docs:` 文档更新
- `refactor:` 代码重构
- `test:` 测试相关

## 4、团队协作工具和实践

### 4.1 代码审查（Code Review）

代码审查是提高代码质量、分享知识和保持代码风格一致的关键实践。

**游戏开发代码审查要点**：
1. **功能性**：代码是否按预期工作？
2. **性能**：是否有潜在的性能问题？
3. **可维护性**：代码是否清晰易懂？
4. **一致性**：是否符合项目代码规范？
5. **测试**：是否有相应的测试覆盖？

**工具推荐**：
- **GitHub Pull Requests**：开源项目首选
- **GitLab Merge Requests**：企业自托管方案
- **Phabricator**：Facebook开源，功能全面

### 4.2 Issue和项目管理

将任务、Bug、功能请求等管理工作与代码仓库集成。

**GitHub Issues最佳实践**：
- 使用模板保证信息完整
- 标签分类（bug, enhancement, art, sound等）
- 里程碑跟踪版本进度
- 看板板可视化工作流

**游戏开发Issue模板示例**：
```markdown
## 问题描述

**游戏版本**：v1.2.3
**平台**：Windows 10
**发生频率**：每次/偶尔/特定情况

## 重现步骤
1. 打开游戏
2. 进入主菜单
3. 点击"新游戏"
4. 观察崩溃

## 预期行为
正常进入游戏场景

## 实际行为
游戏崩溃，显示错误信息

## 附加信息
截图、错误日志、系统信息等
```

### 4.3 持续集成/持续部署（CI/CD）

自动化构建、测试和部署流程，提高开发效率。

**游戏开发CI/CD特殊考虑**：
- **构建时间长**：需要缓存机制
- **平台多样**：Windows、macOS、Linux、移动端
- **资产处理**：需要处理美术资源导入
- **性能测试**：需要自动化性能基准测试

**GitHub Actions配置示例**：
```yaml
name: Unity Build

on:
  push:
    branches: [ main, develop ]
  pull_request:
    branches: [ main ]

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
    - uses: actions/checkout@v3
      with:
        lfs: true

    - name: Unity Build
      uses: game-ci/unity-builder@v2
      with:
        targetPlatform: StandaloneWindows64

    - name: Upload Build Artifact
      uses: actions/upload-artifact@v3
      with:
        name: WindowsBuild
        path: build
```

## 5、游戏资产版本管理

### 5.1 美术资源管理

美术资源（纹理、模型、动画）是游戏开发中最难版本控制的部分。

**最佳实践**：
1. **源文件与导入文件分离**
   - 存储`.psd`, `.blend`, `.ai`等源文件
   - 不存储引擎导入后的中间文件
   - 使用脚本自动化导入流程

2. **资产命名规范**
   ```
   角色/主角/模型/alice.fbx
   角色/主角/纹理/alice_diffuse.png
   角色/主角/纹理/alice_normal.png
   角色/主角/动画/idle.fbx
   ```

3. **版本化预览图**
   - 为每个重要资产生成预览图
   - 在Pull Request中显示差异对比

### 5.2 场景和预制件协作

多人同时编辑场景和预制件容易导致冲突。

**解决方案**：
- **场景分块**：将大场景拆分为多个子场景
- **预制件化**：将可重用部分制作成预制件
- **锁定机制**：使用工具标记正在编辑的资产
- **频繁合并**：小步快跑，减少冲突范围

### 5.3 配置和数据文件

游戏配置、平衡数据等需要特别关注。

**JSON/XML/YAML文件管理**：
- 使用易读的格式（优先JSON）
- 避免存储自动生成的ID
- 使用合并友好的格式（每行一个对象）
- 为数值变化添加注释说明原因

## 6、常见问题与解决方案

### 6.1 合并冲突解决

**二进制文件冲突**：
```bash
# 使用"我们的"版本
git checkout --ours 文件名.psd

# 使用"他们的"版本
git checkout --theirs 文件名.psd

# 手动解决后标记为已解决
git add 文件名.psd
```

**场景文件冲突**：
1. 备份两个版本的文件
2. 使用引擎工具合并（如Unity的Smart Merge）
3. 手动检查重要变化
4. 测试合并后的场景

### 6.2 仓库清理和维护

**清理历史大文件**：
```bash
# 查找大文件
git lfs ls-files
git rev-list --objects --all | git cat-file --batch-check='%(objecttype) %(objectname) %(objectsize) %(rest)' | awk '/^blob/ {print substr($0,6)}' | sort --numeric-sort --key=2 | tail -10

# 使用BFG工具清理
java -jar bfg.jar --strip-blobs-bigger-than 100M 仓库目录
```

**优化仓库性能**：
```bash
# 垃圾回收
git gc --aggressive --prune=now

# 重新打包
git repack -a -d --depth=250 --window=250
```

### 6.3 备份策略

**3-2-1备份原则**：
- **3**份数据副本
- **2**种不同存储介质
- **1**份离线备份

**游戏项目备份计划**：
1. **本地仓库**：开发电脑上的工作副本
2. **远程仓库**：GitHub/GitLab/Gitee等云服务
3. **本地备份**：定期完整备份到外部硬盘
4. **云端备份**：使用Backblaze等云备份服务

## 总结

版本控制是游戏开发专业化的标志。对于独立开发者和小团队，掌握Git及其相关工具不仅能防止数据丢失，更能显著提高开发效率和协作质量。

**核心要点回顾**：

1. **早用版本控制**：项目第一天就应该设置版本控制
2. **小步频繁提交**：每次完成一个小功能就提交，描述清晰
3. **善用分支**：每个新功能、Bug修复都在独立分支进行
4. **处理大文件**：使用Git LFS管理美术资源等大文件
5. **自动化一切**：通过CI/CD自动化构建、测试、部署
6. **定期备份**：远程仓库不是备份的全部，要有完整备份策略

**给独立开发者的最小可行方案**：
1. 在GitHub/GitLab创建免费仓库
2. 安装配置Git LFS
3. 设置合适的.gitignore
4. 每天结束工作前提交所有更改
5. 每周备份整个项目到外部硬盘

记住：好的版本控制习惯就像游戏中的存档点——你可能不经常需要它们，但当需要时，你会非常庆幸它们存在。

## 动手实践

1. **Git基础练习**：
   - 在GitHub创建新仓库
   - 克隆到本地，添加简单的Unity/Godot项目
   - 练习add、commit、push、pull基本操作
   - 创建并合并一个功能分支

2. **Git LFS配置**：
   - 在现有项目中安装配置Git LFS
   - 添加几个大文件（可创建测试用的PSD或FBX文件）
   - 观察.gitattributes文件变化
   - 推送并验证LFS文件是否正确处理

3. **协作模拟**：
   - 找一个伙伴或使用两个本地目录模拟协作
   - 同时修改同一个文件，产生冲突
   - 练习冲突解决流程
   - 使用Pull Request流程进行代码审查

4. **CI/CD体验**：
   - 在GitHub仓库中启用GitHub Actions
   - 配置简单的自动构建工作流
   - 观察自动构建过程和结果
   - 尝试添加自动化测试步骤
