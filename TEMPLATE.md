# Markdown写作模板

## 概述

本文档是《独立游戏开发手册》的写作模板和规范指南。所有手册内容应遵循此模板，确保一致性、可读性和维护性。

## 文件命名规范

### 目录命名
- 使用英文名称，数字前缀（如`01Preparation`）
- 使用首字母大写，单词间用空格分隔（如`Game Design Foundation`）
- 保持简洁、描述性

### 文件命名
- 使用中文名称，数字前缀（如`01-心态与目标设定.md`）
- 格式：`数字-主题.md`（如`02-创意与灵感来源.md`）
- 避免特殊字符，使用连字符`-`分隔单词
- 文件名应准确反映内容主题

## 内容编写指南

### 语言风格
1. **浅显易懂**：避免过度专业术语，必要术语需提供通俗解释
2. **指导性强**：提供具体步骤、可操作建议、实际案例
3. **鼓励性语气**：理解独立开发的困难，给予正向激励
4. **中英对照**：关键术语提供英文原文（如`独立游戏（Indie Game）`）

### 技术内容处理
1. **代码示例**：
   ```python
   # 提供完整、可运行的代码示例
   def example_function():
       return "Hello, World!"
   ```

2. **命令示例**：
   ```bash
   # 命令行示例
   git commit -m "提交信息"
   ```

3. **配置示例**：
   ```json
   {
     "key": "value",
     "array": [1, 2, 3]
   }
   ```

### 视觉元素
1. **表格**：用于对比、列出选项等
   | 特征 | 选项A | 选项B |
   |------|-------|-------|
   | 优点 | 优点1 | 优点2 |
   | 缺点 | 缺点1 | 缺点2 |

2. **列表**：
   - 无序列表用于列举项目
   - 有序列表用于步骤说明
   - 嵌套列表用于层次结构

3. **强调**：
   - **加粗**：强调关键概念
   - *斜体*：表示术语、引用
   - `代码样式`：表示代码、命令、文件名

## 章节组织原则

### 渐进式学习
1. **从基础到高级**：每个章节内部应有逻辑递进
2. **概念先于实践**：先解释概念，再提供实践指导
3. **示例驱动**：每个重要概念应有具体游戏示例

### 内容深度控制
1. **核心内容**：确保覆盖该主题的核心知识点
2. **延伸内容**：可选的高级主题或深入探讨
3. **参考资料**：提供进一步学习的资源链接

### 交叉引用
1. **内部引用**：引用本手册其他章节内容
   ```
   [相关章节标题](../路径/文件名.md)
   ```

2. **外部引用**：引用权威外部资源
   ```
   [资源标题](https://example.com)
   ```

## 元数据规范

### 文档头部
- **标题**：使用一级标题（`#`）
- **概述**：简要说明本章内容和目标
- **学习目标**：明确列出读者学完本章后的收获

### 文档尾部
- **最后更新**：文档最后修订日期（格式：YYYY-MM-DD）
- **相关章节**：链接到相关内容章节
- **前置知识**：学习本章前需要掌握的知识（如适用）

## 特殊内容处理

### 警告和注意
> **注意**：重要但容易忽略的细节
>
> **警告**：可能导致问题的操作或常见错误

### 提示和建议
> **提示**：提高效率的技巧或建议
>
> **最佳实践**：经过验证的有效方法

### 示例和案例
> **示例**：具体的应用场景示例
>
> **案例研究**：真实项目的分析

## 图片和图表

### 图片使用
1. **存储位置**：图片存储在章节目录的`images/`子目录
2. **命名规范**：`主题-描述.png`（如`engine-comparison.png`）
3. **引用格式**：
   ```
   ![图片描述](./images/文件名.png)
   ```

### 图表规范
1. **清晰可读**：确保在多种设备上清晰可读
2. **颜色对比**：使用足够的颜色对比度
3. **文字大小**：图表文字应足够大

## 示例文档

### 完整示例
以下是一个完整的文档示例：

```markdown
# 游戏引擎选择指南

## 概述

本章将介绍如何为独立游戏项目选择合适的游戏引擎。我们将对比主流引擎的特点、适用场景和学习曲线，帮助开发者做出明智选择。

## 1、引擎选择因素

### 1.1 项目需求分析
- **游戏类型**：2D、3D、VR、移动端等
- **团队规模**：单人开发、小团队、协作需求
- **目标平台**：PC、移动端、主机、Web
- **技术复杂度**：物理模拟、网络功能、AI需求

### 1.2 团队技能评估
- **编程经验**：团队成员熟悉的编程语言
- **美术能力**：美术风格和技术需求
- **学习能力**：学习新工具的时间和意愿

## 2、主流引擎对比

### 2.1 Unity
**优点**：
- 强大的生态系统和社区支持
- 跨平台支持优秀
- 丰富的资产商店资源

**缺点**：
- 性能优化需要更多工作
- 2D功能相对基础

### 2.2 Godot
**优点**：
- 完全开源，无版权费用
- 轻量级，运行效率高
- 内置节点系统直观易用

**缺点**：
- 第三方资源相对较少
- 大规模项目支持有限

（更多引擎对比...）

## 总结

选择游戏引擎是独立游戏开发的关键决策。没有"最佳"引擎，只有"最适合"的引擎。考虑项目需求、团队技能、长期维护等因素，选择最能支持你实现创意的工具。
```

## 常见问题

### Q: 如何确定文档的详细程度？
A: 以初学者能理解为目标。如果概念复杂，先提供简单解释，再提供深入探讨或延伸阅读链接。

### Q: 如何处理过时内容？
A: 定期审查技术内容。如果内容过时，更新日期并添加兼容性说明。重大变更可创建新版本文档。

### Q: 如何平衡中文和英文术语？
A: 首次出现的关键术语使用"中文（英文）"格式，后续可使用中文。在术语表或延伸阅读中提供英文资源链接。

### Q: 如何添加交互式内容？
A: 当前版本仅支持静态Markdown。未来可考虑添加：
- 可运行的代码示例
- 交互式图表
- 在线工具链接
