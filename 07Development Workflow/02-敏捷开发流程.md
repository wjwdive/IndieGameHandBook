# 敏捷开发流程

## 概述

敏捷开发（Agile Development）是一种以迭代和增量为核心的软件开发方法论，特别适合需求变化频繁的独立游戏项目。本章介绍敏捷原则在游戏开发中的应用，包括Scrum和Kanban等框架，以及如何建立高效的迭代工作流程。

## 1、敏捷开发基础

### 1.1 敏捷宣言与原则
**敏捷宣言四大价值**：
1. **个体和互动**高于流程和工具
2. **可工作的软件**高于详尽的文档
3. **客户合作**高于合同谈判
4. **响应变化**高于遵循计划

**敏捷十二原则**（游戏开发视角）：
- 通过早期和持续交付有价值的游戏内容使玩家满意
- 欢迎需求变化，即使开发后期（玩家反馈驱动的改进）
- 频繁交付可工作的游戏版本（每周/每两周）
- 业务人员与开发者每日协同工作
- 围绕有动力的个体构建项目，给予所需支持
- 面对面沟通是最有效的信息传递方式
- 可工作的游戏是进度的首要度量标准
- 敏捷过程促进可持续发展（避免加班文化）
- 持续关注技术卓越和良好设计
- 简洁性——最大化未完成工作量的艺术——至关重要
- 自组织团队产生最佳架构、需求和设计
- 定期反思如何更有效，并相应调整行为

### 1.2 为什么游戏开发需要敏捷
**游戏开发的特点**：
- **高度创造性**：设计在不断演变
- **不确定性高**：玩家反应难以预测
- **跨学科协作**：策划、程序、美术、音频紧密配合
- **迭代本质**：游戏体验通过反复试玩优化

**瀑布模型的局限性**：
- 前期规划难以预见所有问题
- 变更成本高昂
- 反馈来得太晚
- 难以适应市场变化

**敏捷的优势**：
- 快速响应玩家反馈
- 早期发现问题
- 保持团队动力和灵活性
- 降低整体风险

### 1.3 敏捷框架选择
**Scrum框架**：
- **特点**：时间盒迭代（Sprint）、固定角色、仪式化会议
- **适用**：中小团队、明确目标的项目、需要结构化的团队

**Kanban方法**：
- **特点**：可视化工作流、限制在制品、持续交付
- **适用**：维护期项目、内容生产、支持工作

**混合方法**：
- Scrumban：结合Scrum的迭代和Kanban的可视化
- 自定义流程：根据团队需求调整的敏捷实践

## 2、Scrum在游戏开发中的应用

### 2.1 Scrum核心元素
**三种角色**：
1. **产品负责人（Product Owner）**：
   - 代表玩家和商业利益
   - 维护产品待办列表（Product Backlog）
   - 确定功能优先级
   - 接受或拒绝工作成果

2. **Scrum Master**：
   - 确保团队遵循Scrum流程
   - 移除障碍，保护团队
   - 促进团队自组织和改进
   - 不一定是管理者，而是服务型领导

3. **开发团队**：
   - 跨职能团队（策划、程序、美术、音频）
   - 自组织，决定如何完成工作
   - 通常5-9人规模
   - 共同对交付负责

**三种工件**：
1. **产品待办列表（Product Backlog）**：
   - 所有需要完成的工作项有序列表
   - 包含功能、bug、技术任务等
   - 由产品负责人维护优先级

2. **冲刺待办列表（Sprint Backlog）**：
   - 当前冲刺中选择要完成的工作项
   - 团队承诺在冲刺内完成
   - 冲刺期间可以调整细节

3. **增量（Increment）**：
   - 冲刺结束时产生的可工作游戏版本
   - 必须是"完成完成"（真正可玩）
   - 每个冲刺增加游戏价值

**五种事件**：
1. **冲刺（Sprint）**：
   - 固定时长的时间盒（通常2-4周）
   - 期间不改变目标，保持稳定
   - 产出可发布的游戏增量

2. **冲刺规划（Sprint Planning）**：
   - 确定冲刺目标和选择待办项
   - 通常1-2小时每周冲刺时长
   - 产出冲刺目标和冲刺待办列表

3. **每日站会（Daily Scrum）**：
   - 15分钟同步进度和计划
   - 回答三个问题：昨天做了什么？今天计划做什么？有什么障碍？
   - 不解决问题，只识别问题

4. **冲刺评审（Sprint Review）**：
   - 展示冲刺成果，收集反馈
   - 邀请利益相关者（玩家代表、发行商等）
   - 讨论下一步计划

5. **冲刺回顾（Sprint Retrospective）**：
   - 团队反思和改进流程
   - 讨论：什么做得好？什么可以改进？下一步行动？
   - 持续改进工作方式

### 2.2 游戏开发中的Scrum实践
**游戏特定调整**：
- **冲刺长度**：2周通常适合游戏开发节奏
- **"完成"定义**：包含测试、集成、文档、性能检查
- **冲刺目标**：例如"实现基础战斗系统"而非"完成10个任务"
- **跨职能协作**：确保每个冲刺都有各专业贡献

**冲刺规划技巧**：
- **用户故事格式**：作为[玩家类型]，我想要[功能]，以便[价值]
- **故事点估算**：使用斐波那契数列（1,2,3,5,8,13）
- **容量规划**：考虑假期、会议、支持工作
- **技术债务管理**：分配一定比例时间处理技术债务

**每日站会变体**：
- **物理看板**：团队成员移动任务卡片
- **数字工具**：Trello、Jira、Notion等
- **分布式团队**：使用视频会议，保持简短高效
- **混合团队**：核心成员现场，远程成员视频参与

### 2.3 游戏冲刺示例
**2周冲刺计划**：
```
第1周：
- 周一：冲刺规划，开始核心战斗编程
- 周二：战斗动画制作，音效设计开始
- 周三：敌人AI编程，美术资源优化
- 周四：集成测试，平衡调整
- 周五：内部测试，bug修复

第2周：
- 周一：UI系统开发，视觉效果增强
- 周二：音乐集成，性能优化
- 周三：完整关卡集成，系统测试
- 周四：冲刺评审准备，演示版本制作
- 周五：冲刺评审，冲刺回顾，下一个冲刺规划
```

**冲刺成果**：
- 可玩的战斗演示版本
- 玩家可体验核心战斗循环
- 团队获得早期反馈
- 识别需要改进的问题

## 3、Kanban在游戏开发中的应用

### 3.1 Kanban核心原则
**Kanban四大原则**：
1. **可视化工作流**：让所有工作项和状态可见
2. **限制在制品**：控制同时进行的工作数量
3. **管理工作流**：关注工作的流动效率
4. **明确流程策略**：团队共识的工作规则

**Kanban五大实践**：
1. **可视化**：使用看板展示工作项和流程
2. **限制在制品**：每列设置最大工作项数量
3. **管理流动**：监控和优化工作流动
4. **明确规则**：定义何时可以移动工作项
5. **改进协作**：基于数据持续改进

### 3.2 游戏开发看板设计
**典型看板列**：
```
待办 → 分析/设计 → 开发 → 测试 → 集成 → 完成
```

**游戏开发特定列**：
- **概念**：新想法和功能建议
- **设计**：游戏设计文档、原型设计
- **美术生产**：2D/3D资源制作
- **程序开发**：编码实现
- **音频制作**：音乐和音效创作
- **QA测试**：质量保证测试
- **玩家测试**：外部玩家反馈
- **发布准备**：打包和部署

**工作项类型**：
- **功能**：新游戏功能或系统
- **Bug**：错误修复
- **技术债务**：代码重构和优化
- **内容**：关卡、角色、物品等
- **维护**：工具和流程改进

### 3.3 在制品限制与流动优化
**设置WIP限制**：
- **开发列**：团队开发者数量 × 1.5
- **测试列**：测试人员数量 × 2
- **设计列**：设计师数量 × 2
- **总WIP**：团队总人数 × 2（避免过度并行）

**流动度量**：
- **周期时间**：从开始到完成的平均时间
- **吞吐量**：单位时间完成的工作项数量
- **累积流图**：显示各状态工作项数量变化
- **瓶颈识别**：哪一列经常达到WIP限制

**改进措施**：
- **解决瓶颈**：为瓶颈环节增加资源或改进流程
- **减少批处理**：小批量工作，快速流动
- **平衡工作类型**：混合功能、bug、技术债务
- **定期审查**：每周审查流动数据，调整流程

## 4、敏捷实践与工具

### 4.1 用户故事与任务分解
**优秀用户故事特征**：
- **独立的**：可独立实现和测试
- **可协商的**：细节可在实现时讨论
- **有价值的**：为玩家提供明确价值
- **可估算的**：团队能估算工作量
- **小规模的**：适合在一个冲刺内完成
- **可测试的**：有明确的验收标准

**游戏开发用户故事示例**：
```
作为动作游戏玩家，我想要连招系统，以便通过技能组合创造个性化战斗风格。
验收标准：
- 玩家可通过按键组合触发不同连招
- 连招有视觉效果和音效反馈
- 不同武器有不同连招组合
- 连招造成额外伤害或效果
```

**任务分解技巧**：
- **技术任务**：编程、集成、优化
- **美术任务**：概念、建模、动画、UI
- **设计任务**：数值平衡、关卡设计、叙事
- **音频任务**：音乐制作、音效设计、语音
- **测试任务**：单元测试、集成测试、用户测试

### 4.2 估算与规划
**估算方法**：
- **故事点**：相对复杂度，使用斐波那契数列
- **理想时间**：无干扰情况下所需时间
- **计划扑克**：团队匿名投票，讨论差异
- **亲和估算**：将用户故事分组为复杂度级别

**速度跟踪**：
- **速度**：团队一个冲刺能完成的故事点总数
- **使用方式**：预测未来冲刺能完成的工作量
- **注意事项**：速度是趋势，不是承诺
- **影响因素**：假期、成员变动、技术债务

**发布规划**：
- **基于速度的预测**：剩余故事点 ÷ 平均速度 = 冲刺数量
- **信心区间**：使用最小/平均/最大速度预测范围
- **里程碑规划**：将发布分解为多个可验证的里程碑
- **缓冲区**：为不确定性和意外情况预留时间

### 4.3 敏捷工具栈
**看板工具**：
- **Trello**：简单直观，适合小型团队
- **Jira**：功能强大，适合复杂项目
- **Azure DevOps**：集成完整开发工具链
- **Notion**：灵活自定义，适合创意工作

**协作工具**：
- **Slack/Discord**：实时沟通和集成
- **Miro/Mural**：远程头脑风暴和规划
- **Google Workspace**：文档共享和协作
- **GitHub/GitLab**：代码协作和项目管理

**游戏特定工具**：
- **Perforce Helix Core**：大文件版本控制
- **ShotGrid**：游戏生产管理
- **Hansoft**：游戏项目组合管理
- **自定义工具**：根据团队需求构建

## 5、敏捷团队建设

### 5.1 自组织团队培养
**自组织特征**：
- **共同目标**：团队对游戏愿景有共识
- **技能多样性**：成员互补，能解决各种问题
- **决策自主**：团队决定如何完成工作
- **持续学习**：积极改进技能和流程
- **相互问责**：成员对团队成果共同负责

**促进自组织的领导行为**：
- **授权**：给予团队决策空间
- **支持**：提供资源和移除障碍
- **信任**：相信团队能力和判断
- **反馈**：提供建设性反馈，帮助成长
- **榜样**：展示期望的行为和态度

### 5.2 跨职能协作
**游戏开发跨职能挑战**：
- **专业语言差异**：策划、程序、美术术语不同
- **工作节奏差异**：程序迭代快，美术生产慢
- **反馈循环**：早期和频繁的跨专业反馈
- **共同理解**：对齐游戏愿景和设计意图

**改进协作的策略**：
- **结对工作**：程序与美术结对解决技术美术问题
- **定期同步**：跨专业站会或同步会议
- **原型驱动**：快速原型促进早期反馈
- **共享语言**：建立团队术语表，减少误解
- **共同成功标准**：定义什么是"完成"的共同标准

### 5.3 远程与混合团队敏捷
**远程敏捷挑战**：
- **沟通延迟**：非同步沟通降低效率
- **文化差异**：时区、语言、工作习惯不同
- **信任建立**：缺乏面对面互动影响信任
- **工具依赖**：高度依赖数字工具和流程

**远程敏捷最佳实践**：
- **重叠工作时间**：确保核心时段同步沟通
- **视频优先**：复杂讨论使用视频会议
- **文档文化**：重要决策和讨论书面记录
- **虚拟社交**：定期非工作互动建立关系
- **工具标准化**：统一协作工具和流程

## 6、敏捷度量与改进

### 6.1 敏捷度量指标
**团队健康指标**：
- **冲刺目标达成率**：是否完成承诺的目标
- **团队幸福感**：定期调查团队满意度
- **离职率**：团队成员流动情况
- **可持续性**：加班情况和工作生活平衡

**过程效率指标**：
- **速度趋势**：团队交付能力的变化
- **周期时间**：从开始到完成的平均时间
- **流动效率**：工作时间占总时间的比例
- **在制品水平**：同时进行的工作数量

**质量指标**：
- **缺陷密度**：每千行代码或每个功能的缺陷数
- **逃逸缺陷**：发布后发现的缺陷比例
- **技术债务比例**：技术债务工作占总工作比例
- **自动化测试覆盖率**：代码被自动化测试覆盖的比例

### 6.2 冲刺回顾实践
**有效回顾要素**：
- **心理安全**：成员能坦诚发言而不担心后果
- **数据驱动**：基于实际数据而非主观感受
- **行动导向**：产生具体的改进行动
- **跟踪落实**：检查上次回顾行动的进展

**回顾技巧**：
- **星型回顾**：继续做什么？开始做什么？停止做什么？
- **时间线回顾**：按时间顺序回顾冲刺事件
- **开心-不开回顾**：列出开心和不开心的时刻
- **5个为什么**：深入分析问题的根本原因

**改进行动示例**：
- **流程改进**：调整每日站会时间或格式
- **工具改进**：引入新工具或优化现有工具
- **技能提升**：组织技术分享或培训
- **协作改进**：建立新的沟通渠道或仪式

### 6.3 规模化敏捷考虑
**多团队协调**：
- **Scrum of Scrums**：各团队代表定期同步
- **产品负责人团队**：协调不同团队的产品负责人
- **发布火车**：固定节奏的跨团队集成和发布
- **社区实践**：建立实践社区分享最佳实践

**大规模游戏项目**：
- **分层规划**：史诗→特性→用户故事→任务
- **集成节奏**：定期集成各团队工作成果
- **架构协调**：确保技术架构的一致性和兼容性
- **文化一致性**：保持敏捷价值观和原则的统一

## 7、敏捷在独立游戏中的挑战与适应

### 7.1 单人开发者敏捷
**单人敏捷挑战**：
- 缺乏团队协作和相互督促
- 难以保持客观和自我批评
- 容易陷入细节而失去大局
- 缺乏跨职能视角

**单人敏捷实践**：
- **个人看板**：可视化工作，限制在制品
- **时间盒冲刺**：设定固定周期目标
- **外部反馈**：定期向玩家或同行展示进展
- **结对编程**：偶尔与其他开发者结对工作
- **虚拟团队**：在线社区中获得支持和反馈

### 7.2 资源限制下的敏捷
**小团队限制**：
- 成员身兼多职，上下文切换频繁
- 难以保持所有敏捷仪式
- 工具和流程需要轻量化
- 学习曲线和过程开销有限制

**轻量级敏捷**：
- **简化仪式**：每日站会改为每周两次，回顾会简化
- **简化工件**：使用简单待办列表，减少文档
- **工具最小化**：选择免费或低成本工具
- **混合方法**：结合最适合团队的部分实践
- **逐步引入**：从1-2个实践开始，逐步增加

### 7.3 创意与结构的平衡
**过度流程化风险**：
- 流程取代思考，机械执行
- 创造性工作被僵化框架限制
- 官僚主义消耗创作能量
- 游戏变成功能列表，失去灵魂

**平衡策略**：
- **原则而非规则**：理解敏捷原则，灵活应用实践
- **定期检查价值**：每个实践是否真正增加价值？
- **游戏优先**：所有流程服务于创造优秀游戏
- **团队定制**：根据团队特点和项目需求调整流程
- **持续改进**：不断优化流程，保持高效和愉快

## 总结

敏捷开发不是一套固定规则，而是一种思维方式和响应变化的能力。在独立游戏开发中应用敏捷的关键是：

1. **价值导向**：始终关注为玩家创造价值，而非机械完成工作
2. **持续学习**：通过快速迭代和反馈不断学习和改进
3. **团队协作**：建立高效协作的跨职能团队
4. **灵活适应**：根据项目需求和团队特点调整方法
5. **平衡创意与效率**：流程服务于创意，而非限制创意

记住，最好的流程是让你的团队能够高效创造优秀游戏的流程。不要害怕尝试、调整和放弃不适合的实践。敏捷的最终目标是帮助你和你的团队更愉快、更有效地创造令玩家喜爱的游戏。

## 动手实践

1. **个人看板设置**：为你的当前项目创建一个Kanban看板（使用Trello或物理白板），至少包含"待办"、"进行中"、"完成"三列，并添加5个当前工作项。

2. **冲刺规划模拟**：选择一个游戏功能，将其分解为3-5个用户故事，为每个故事估算故事点，并规划一个2周的冲刺计划。

3. **回顾会练习**：回顾过去一周的工作，使用"星型回顾"方法（继续/开始/停止），制定3个具体的改进行动。

4. **跨职能沟通**：如果你是程序员，与一位美术或策划讨论一个功能需求，练习使用用户故事格式描述需求，并确保双方理解一致。

5. **流程优化**：分析你当前工作流程中的一个瓶颈，设计一个实验来改进它，实施一周后评估效果。
